# Stage 02 练习

## 练习原则

每个练习都应该让某个边界更安全或更清楚。

不要做对运行时代码没有帮助的抽象类型谜题。

## 练习 1 - 把一个 JS 工具改成 strict TypeScript

目标：在真实代码里感受编译器压力。

任务：

- 选一个 Stage 01 工具
- 加上 `tsconfig.json` 并启用 strict 设置
- 把工具改写成 TypeScript
- 去掉隐式 `any` 和含糊的返回类型

通过标准：

- 改写后的版本比 JS 更容易 review，而不是更难读

## 练习 2 - 建模一个 CLI 契约

目标：把命令行输入输出表达清楚。

任务：

- 为解析后的 CLI 参数定义类型
- 为成功 / 失败结果定义结果类型
- 把主执行路径从头到尾类型化

通过标准：

- 你能指出哪些无效状态因为类型设计被提前收缩了

## 练习 3 - Union 与收窄练习

目标：把条件分支写得显式而安全。

任务：

- 为三种命令执行结果定义判别联合
- 写出覆盖全部分支的处理逻辑
- 加一个 exhaustiveness check 或等价模式

通过标准：

- 你能解释编译器是如何帮助你发现漏分支的

## 练习 4 - 用 `unknown` 替换 `any`

目标：不要在边界绕过类型安全。

任务：

- 选一个接受松散输入的函数
- 把输入边界从 `any` 改成 `unknown`
- 增加 narrowing 函数或 schema 校验
- 比较前后可读性

通过标准：

- 你能解释为什么对外部数据来说，`unknown` 比 `any` 更安全

## 练习 5 - 泛型 helper 练习

目标：练习有用但不过度的泛型。

任务：

- 写一个泛型 `mapValues` 或类似 helper
- 写一个泛型结果包装器，用于命令执行或 API 调用
- 避免不必要的复杂约束

通过标准：

- 这个泛型版本对第一次看到代码的人仍然可理解

## 练习 6 - 运行时校验边界

目标：把编译期信念与运行时真相分开。

任务：

- 从文件或 API 读取 JSON
- 用 schema 验证它
- 把验证后的数据转换成内部类型对象
- 对非法数据报出清楚错误

通过标准：

- 在输入边界上，没有用类型断言替代真正的校验

## 练习 7 - 阅读一段真实 TS 代码

目标：学会读真实世界中的 TypeScript。

任务：

- 找一个中小型后端工具或 SDK 里的 TS 模块
- 标出它的公共类型、内部类型和边界设计
- 写一段评价：哪里因为类型更清楚，哪里因为类型更难读

通过标准：

- 你能批判性地看 TS 设计，而不是只会照抄

## 最低完成要求

至少完成：

- 练习 1、2、3、6
- 把一个已有 JS 工具改成 strict TS
- 写一段对真实 TS 模块的评估说明
